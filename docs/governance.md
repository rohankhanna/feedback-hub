# Governance

## Purpose
Governance keeps shared learnings more trustworthy than raw project notes.

## Authority Boundaries
- projects own their project feedback writes
- curated learnings are promoted through explicit manager-controlled workflows
- projects read curated learnings but do not write to them directly

## Promotion Standard
Promote an artifact only when it is:
- understandable outside its original task
- reusable beyond one narrow situation
- stripped of unnecessary project-specific detail

## Auditability
Curated changes should be traceable back to their originating project feedback artifact.

## Operating Default
- raw project feedback is easier to write and may be noisy
- curated learnings are fewer, cleaner, and more deliberate
- curation should prefer quality over volume

## Public Shell Rule
The public shell describes the governance model generically.
It does not expose private project inventory or internal operating detail.

## Public Disclosure
Public disclosure in this repo lives in the public-facing docs rather than a dedicated `AUTHORSHIP.md` file on `main`.

That disclosure must still cover:
- human stewardship
- AI assistance where material
- the rule that public-bound work is human-reviewed before publication
